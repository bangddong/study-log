처음엔 그냥 해볼 생각이었습니다.


이직 준비가 지겨워서, 차라리 이 과정을 RPG 퀘스트로 만들면 재미있겠다 싶었습니다. Claude Code로 빠르게 뚝딱 만들어보려고 시작했는데, 어느새 두 달이 지나 있었고 PR이 69개가 됐습니다. 그리고 그 과정에서 AI 에이전트를 **어떻게 믿고 맡기는가**에 대해 꽤 많은 것을 배웠습니다.


---


## 문제는 AI가 '너무 잘한다'는 것입니다


처음에는 Claude에게 대화 형태로 기능을 만들어달라고 했습니다. 그런데 AI는 기대보다 훨씬 더 많은 일을 합니다. BE 기능을 구현해달라고 하면 슬그머니 FE 파일도 건드립니다. 리팩토링 해달라고 하면 요청하지 않은 파일에 주석을 달기 시작합니다. 테스트를 작성해달라고 하면 원본 코드까지 수정합니다.


이건 AI가 나쁜 게 아닙니다. **경계가 없으니 경계를 만들지 않는 것**입니다.


사람 팀에서도 마찬가지입니다. 역할이 불명확한 팀에서 일한 적 있으시다면 아실 겁니다. 누구나 모든 파일을 건드리고, 의도 없이 충돌이 생기고, 나중엔 뭘 누가 왜 바꿨는지 아무도 모릅니다.


그래서 **하네스(harness)** 를 짜기 시작했습니다.


---


## 하네스란 무엇인가


하네스는 에이전트에게 **역할 경계, 실행 순서, 물리적 제한**을 주는 구조입니다. 프롬프트로 "BE만 건드려"라고 부탁하는 게 아니라, BE가 아닌 파일을 수정하려고 하면 아예 툴 실행을 막아버리는 것입니다.


이 프로젝트에서 만든 구조는 이렇습니다:


```javascript
.claude/
├── agents/
│   ├── be-feature-builder.md   # BE 전담, fe/ 읽기도 금지
│   ├── fe-feature-builder.md   # FE 전담, BE 스펙 수신 후 구현
│   ├── qa-reviewer.md          # 읽기 전용, 코드 수정 금지
│   ├── logic-reviewer.md
│   ├── convention-reviewer.md
│   └── test-writer.md
├── skills/
│   └── feature-dev/SKILL.md   # 오케스트레이터 스킬
└── scripts/
    ├── assert-be-path.sh       # be/ 외 Write/Edit 시 차단
    └── assert-not-main.sh      # main 브랜치 직접 수정 차단
```


`be-feature-builder` 에이전트의 역할 경계 정의는 이렇게 생겼습니다:


| 허용             | 금지                |
| -------------- | ----------------- |
| be/ 디렉토리 전체    | fe/ 디렉토리 — 읽기도 금지 |
| BE 구현, 빌드, 테스트 | QA 리뷰, FE 코드 판단   |
| API 스펙 반환      | FE 구현 시작, PR 외 작업 |


그리고 PreToolUse hook에서 `assert-be-path.sh`가 Write/Edit 시도를 감시합니다. `fe/` 파일을 건드리려 하면 툴 호출 자체가 차단됩니다.


**규칙을 글로 쓰는 게 아니라, 위반 자체를 불가능하게 만드는 것입니다.**


---


## 파이프라인: 순서가 중요합니다


기능 하나를 구현할 때 에이전트는 순서대로 실행됩니다.


```javascript
1. be-feature-builder  →  BE 구현 완료 + API 스펙 반환
        ↓
2. fe-feature-builder  →  BE 스펙을 받아 FE 구현
        ↓
3. qa-reviewer         →  코드 읽기만 하고 보고서 반환
```


특히 `qa-reviewer`의 설계가 인상적으로 작동했습니다. 이 에이전트는 **구현 의도를 전달받지 않습니다.** 오케스트레이터가 어떤 기능을 왜 만들었는지 설명하지 않고, 그냥 코드만 던져줍니다.


그래서 QA reviewer는 코드만 보고 BE ↔ FE 계약 정합성을 판단합니다. BE의 `ResumeResult.kt`에 `overallScore: Int`가 있는데 FE의 `types/api.types.ts`에 `overall_score: number`로 되어 있으면 — 의도가 어떻든 간에 — 불일치로 리포팅합니다.


이게 왜 중요하냐면, 구현자는 "당연히 맞겠지"라고 생각하기 때문입니다. 만든 사람은 틀린 걸 못 봅니다.

## 부딪혔던 문제들


자동화를 열심히 짜면 자동화 자체가 장애물이 됩니다. 몇 가지 사례입니다.


### Copilot Gate가 머지를 막았습니다


GitHub Copilot 리뷰 코멘트에 답글을 달지 않으면 머지를 블록하는 Gate를 만들었습니다. 그런데 이게 제대로 작동을 안 했습니다.


원인은 `workflow_dispatch`로 실행한 check run이 PR의 `check_suite`에 귀속되지 않는 것이었습니다. GitHub Actions에서 `workflow_dispatch` 트리거로 실행하면 그 결과가 PR 체크 목록에 잡히지 않습니다. `mergeStateStatus`가 계속 `BLOCKED`였고, `--admin`으로만 머지가 가능한 상황이 됐습니다.


해결책: check run 방식을 버리고 **Commit Status API**로 교체했습니다. SHA에 직접 `pending / failure / success`를 기록하면 check_suite 귀속 문제 없이 브랜치 보호 규칙에서 요구 조건으로 등록할 수 있습니다.


PR을 4개 태워가며 찾은 원인이었습니다 (#41, #42, #43, #44).


### OAuth 로그인이 프로덕션에서만 터졌습니다


GitHub OAuth 토큰 교환 요청에 `redirect_uri`를 포함하지 않으면 GitHub이 `DEFAULT` 에러를 반환합니다. 로컬에서는 잘 됐습니다. GitHub OAuth App에 callback URL이 하나뿐이라 문제가 없었기 때문입니다. 프로덕션 배포 후, callback URL이 두 개 이상 등록된 상태에서 `redirect_uri` 없이 요청하면 GitHub가 어느 URL을 써야 할지 몰라 에러를 냅니다.


교훈: **로컬 성공은 프로덕션 성공이 아닙니다.** 특히 OAuth처럼 외부 서비스와 연동하는 부분은 프로덕션 환경 그대로 테스트해야 합니다.


### Sentry가 Spring Boot 4.x를 지원하지 않았습니다


Observability를 위해 Sentry를 붙이려고 했습니다. 의존성 추가하고 설정 다 했는데 빌드가 안 됐습니다. Spring Boot 4.x는 아직 Sentry 공식 지원 대상이 아니었습니다. 결국 PR을 머지했다가 (#48) 바로 다음 PR에서 제거했습니다 (#52).


작업 시간을 날린 게 아깝긴 했지만, 이런 실패가 기록으로 남는 것 자체가 가치가 있습니다. 이후에 "왜 Sentry가 없냐"는 질문이 나왔을 때 커밋 히스토리로 답할 수 있습니다.

## 하네스 엔지니어링에서 배운 것


**1. 에이전트는 규칙이 아니라 구조로 통제합니다**


프롬프트에 "FE 파일 건드리지 마"라고 쓰는 건 약합니다. 에이전트가 맥락을 잃거나 긴 작업 중에 흘려버릴 수 있습니다. Hook으로 툴 실행 자체를 막는 게 훨씬 강합니다.


**2. QA는 독립성이 생명입니다**


QA reviewer에게 "이런 기능을 만들었으니 잘 됐는지 확인해"라고 하면 확증 편향이 생깁니다. 구현 의도를 모르는 채로 코드만 보게 하면 만든 사람이 놓친 걸 잡아냅니다.


**3. CI/CD도 에이전트 파이프라인의 일부입니다**


CI가 통과해야 에이전트 작업이 의미 있습니다. Gate를 만들고 브랜치 보호 규칙에 연결하면, 에이전트가 만든 코드도 같은 기준을 통과해야 머지됩니다. 자동화의 신뢰성은 CI에서 옵니다.


**4. 컨텍스트 관리도 설계가 필요합니다**


Claude Code의 컨텍스트 윈도우는 한정돼 있습니다. `CONTEXT.md`가 200줄을 넘어가면 에이전트가 절반쯤은 읽지 않습니다. 활성 컨텍스트(지금 뭘 하고 있나)와 아카이브(이전에 뭘 했나)를 분리해서 관리하면 세션을 이어가면서도 품질이 유지됩니다.


---


## 마치며


혼자 만드는 사이드 프로젝트지만, 하네스를 짜고 나서는 팀처럼 돌아가는 느낌이 생겼습니다. BE 담당자, FE 담당자, QA 담당자가 따로 있고, 서로의 영역을 침범하지 않으며, CI를 통과해야 머지됩니다.


AI를 도구로 쓰는 것과 파이프라인으로 쓰는 것은 다릅니다. 도구는 내가 매번 지시해야 하지만, 파이프라인은 내가 설계한 규칙대로 스스로 흐릅니다.


그 흐름을 만드는 게 하네스 엔지니어링입니다.

Claude Code로 오래 작업하다 보면 어느 순간 "compacting..." 메시지가 뜨고, 그 이후로 Claude가 이전 맥락을 잘 기억하지 못하는 것 같다는 느낌을 받습니다. 마치 대화 중에 갑자기 기억을 잃는 것처럼요.


이건 버그가 아닙니다. **Auto Compaction**이라는 의도적인 메커니즘입니다.


---


## 167,000이라는 숫자


Claude Code는 컨텍스트 토큰 수가 일정 임계값을 넘으면 자동으로 대화를 요약합니다. Sonnet 기준 그 숫자는 **167,000 토큰**입니다.

> 📌 **[2026-03 추가]** Sonnet 4.6, Opus 4.6/4.7는 **1M 컨텍스트 윈도우**를 지원합니다. Max/Team/Enterprise 플랜은 자동 적용, Pro 플랜은 `/extra-usage` 명령으로 opt-in 할 수 있습니다. 1M 윈도우 사용 시 auto-compaction 임계값은 ~950K로 달라집니다. ([출처](https://claude.com/blog/1m-context-ga))

어떻게 계산된 걸까요?


```javascript
200,000  모델 컨텍스트 윈도우
- 20,000  요약 출력 예약 버퍼 (p99.99 통계 기반)
= 180,000  유효 컨텍스트 윈도우
- 13,000  현재 턴 안전 버퍼
= 167,000  자동 압축 임계값 (83.5%)
```


20,000 토큰 예약은 임의의 값이 아닙니다. 실제 압축 요약의 p99.99가 17,387 토큰이라는 통계 데이터에서 나왔습니다. 13,000 버퍼는 현재 턴이 진행되는 중에도 토큰이 계속 쌓이기 때문에 두는 안전 마진입니다.


"최악의 경우도 버텨야 한다"는 설계 원칙이 숫자에 그대로 반영되어 있습니다.

## 압축은 무엇을 기억하고 무엇을 잊는가


압축이 트리거되면 Claude는 9개 섹션으로 구성된 템플릿으로 대화를 요약합니다.


| 섹션     | 잘 보존됨    | 잘 사라짐      |
| ------ | -------- | ---------- |
| 사용자 의도 | 명시적 요청   | 묵시적 기대     |
| 파일/코드  | 변경 파일 목록 | 매우 긴 diff  |
| 에러/수정  | 에러와 해결법  | —          |
| 문제 해결  | 최종 결과    | 거부된 접근과 이유 |
| 현재 작업  | 직전 작업 상태 | —          |


여기서 가장 주목할 것은 **"왜 그걸 하지 않았는가"** 입니다. 템플릿은 무엇을 했는지는 잘 잡아내지만, 무엇을 시도해봤다가 포기했는지는 약합니다. "Redis는 이 프로젝트에 맞지 않아서 쓰지 않기로 했습니다"라고 세션 초반에 말했다면, 여러 번 압축을 거친 후에는 Claude가 다시 Redis를 제안할 수 있습니다.


또 하나는 **초기의 미묘한 선호도**입니다. "default export는 쓰지 마세요"처럼 초반에 한 번 언급한 것들이 압축을 거치면서 희석될 수 있습니다.


---


## 압축 직후, 파일 5개를 다시 읽습니다


압축이 완료되면 Claude Code는 이전에 읽었던 파일 캐시를 클리어하고, **최대 5개 파일, 총 50K 토큰 한도**로 가장 중요한 파일을 다시 주입합니다. `CLAUDE.md`는 시스템 프롬프트에 있어서 이 과정의 영향을 받지 않습니다.


그래서 압축 후 Claude가 "파일을 다시 읽어야 할 것 같습니다"라고 말하는 건 정상 동작입니다.


---


## 3회 실패 후 회로가 끊깁니다


압축 후에도 167K를 초과하는 경우가 있습니다. 시스템 메시지나 첨부 파일 같은 **압축 불가능한 콘텐츠**가 너무 많으면 압축을 해도 임계값 아래로 내려오지 않습니다. 이러면 다음 턴에 즉시 재압축을 시도하고, 또 실패하고, 또 시도하는 루프가 생깁니다.


실제로 이 루프가 한 세션에서 **3,272회** 발생한 사례가 있습니다. 하루에 250,000건의 헛 API 호출이 발생했습니다.


Claude Code는 이를 막기 위해 **회로 차단기(Circuit Breaker)** 를 내장하고 있습니다.


```javascript
MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES = 3
```


연속 3회 실패 시 자동 압축이 완전히 중단됩니다. 이때 수동 `/compact` 명령은 여전히 작동합니다. 그래도 안 되면 새 세션을 시작하는 것이 가장 빠른 복구 방법입니다.


---


## 너무 긴 대화는 압축 자체가 불가능합니다


190K 토큰이 넘는 극단적으로 긴 세션에서는 압축 요청 자체가 API 입력 제한을 초과할 수 있습니다. "압축하려고 보냈더니 그것도 너무 길다"는 상황입니다.


이때 PTL(Prompt Too Long) 재시도 메커니즘이 작동합니다. 가장 오래된 대화부터 잘라내면서 최대 3회 재시도합니다. 3회 모두 실패하면 이런 메시지가 나타납니다.

> Conversation too long. Press esc twice to go up a few messages and try again.

---


## [CLAUDE.md](http://claude.md/)에 압축 지시사항을 남길 수 있습니다


Claude Code는 `CLAUDE.md`에 `## Compact Instructions` 섹션이 있으면 매 압축 시 자동으로 프롬프트에 주입합니다.


```markdown
## Compact Instructions
- 현재 브랜치명, 열린 PR 번호·제목 반드시 보존
- 이 세션에서 시도했다 거부한 접근법과 그 이유
- 발생한 에러와 수정 방법
- 프로젝트 아키텍처 제약 사항
```


압축 후 Claude가 잊기 쉬운 **거부 이유와 초기 제약 사항**을 이 방식으로 살릴 수 있습니다. 중요한 선호도는 `CLAUDE.md` 본문에, 세션별로 기억해야 할 내용은 `## Compact Instructions`에 정리해두는 것이 좋습니다.


---


## 임계값을 직접 조정할 수 있습니다


```bash
# 167K 대신 75%에서 압축 (약 135K)
export CLAUDE_AUTOCOMPACT_PCT_OVERRIDE=75

# 컨텍스트 윈도우 자체를 100K로 제한
export CLAUDE_CODE_AUTO_COMPACT_WINDOW=100000
```


두 환경변수 모두 **축소만 가능**합니다. 167K보다 늦게 압축하도록 설정할 수는 없습니다.

> 📌 **[2026-03 추가]** 1M 컨텍스트 윈도우 사용 시 기본 임계값은 ~95%이므로, 환경변수는 이를 낮추는 용도로 주로 활용됩니다. 200K 기본값 사용자는 기존 설명이 유효합니다. ([출처](https://claudefa.st/blog/guide/mechanics/1m-context-ga))

PTL 에러가 자주 발생하는 긴 세션이라면 `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE=70` 정도로 설정해서 조금 일찍 압축을 유도하는 것이 안전합니다.


---


## 수동 /compact로 타이밍을 직접 제어하세요


자동 압축을 기다리지 말고, 서브태스크 하나가 끝날 때마다 먼저 정리하는 것이 낫습니다. 수동 압축은 커스텀 지시사항을 붙일 수 있다는 장점도 있습니다.


```javascript
/compact 파일 수정 히스토리와 에러 수정 기록 중심으로 압축
```


특히 다음 상황에서는 먼저 압축하는 것이 좋습니다.

- 서브 에이전트 3개 이상 완료 후
- 새 기능 개발 시작 직전 (이전 기능 컨텍스트 불필요)
- 디버깅이 길어진 직후

## 직접 겪은 사례


switch-job-quest 프로젝트에서 오케스트레이터가 be-feature-builder, fe-feature-builder, qa-reviewer를 순차 호출하다 보면 컨텍스트가 생각보다 빨리 찹니다. 문제는 세션 초반에 합의한 내용이 압축 후 사라진다는 점이었습니다.


실제로 세션 초반에 "Kotlin data class 대신 interface로 추상화하지 않기로 했다"고 정했는데, 서브 에이전트 3개가 완료된 후 오케스트레이터가 다시 interface 추상화를 제안하는 상황이 있었습니다. 압축을 거치면서 "무엇을 하지 않기로 했는가"가 사라진 전형적인 케이스입니다.


이후 [CLAUDE.md](http://claude.md/)에 Compact Instructions를 추가했습니다.


```markdown
## Compact Instructions
- 현재 브랜치명과 열린 PR 번호 반드시 보존
- 이 세션에서 시도했다 거부한 접근법과 그 이유
- 서브 에이전트 간 합의된 API 인터페이스 스펙
- 발생한 에러와 수정 방법
```


그리고 서브 에이전트 3개 이상 완료 후에는 다음 에이전트 호출 전에 수동 compact를 먼저 실행합니다.


```javascript
/compact 서브 에이전트 완료 결과 중심으로 압축. 거부된 접근법과 API 스펙 보존.
```


자동 압축을 기다리지 않고 타이밍을 직접 제어하면, 압축 직후 5개 파일 재주입 타이밍도 예측할 수 있어서 오케스트레이터 흐름이 훨씬 안정적입니다.


## 정리


Auto Compaction은 Claude Code가 긴 세션에서도 동작하게 해주는 핵심 메커니즘입니다. 내부 구조를 이해하면 더 잘 활용할 수 있습니다.

- **167K**에서 자동 압축이 트리거됩니다
- **거부 이유와 초기 선호도**가 가장 잘 사라집니다 → `## Compact Instructions`로 보완
- **연속 3회 실패** 시 자동 압축이 중단됩니다 → 수동 `/compact`로 복구
- **환경변수**로 임계값을 당길 수 있습니다 → 긴 세션엔 `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE=70`

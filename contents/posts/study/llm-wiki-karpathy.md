AI에게 같은 것을 반복해서 설명한 적이 있으신가요? 지난주에 공부한 내용, 지난 프로젝트에서 내린 결정, 한 번 정리해둔 개념 — 새 세션을 열 때마다 다시 설명해야 합니다. AI는 매번 처음 만나는 사람처럼 행동합니다.


이것은 프롬프트를 잘못 쓰는 문제가 아닙니다. **컨텍스트가 쌓이지 않기 때문**입니다. 프롬프트 엔지니어링이 "무엇을 어떻게 물어볼 것인가"를 다룬다면, 컨텍스트 엔지니어링은 "AI가 알아야 할 것을 어떻게 구성하고 유지할 것인가"를 다룹니다. LLM Wiki는 그 방법론 중 하나입니다.


Andrej Karpathy가 제안한 이 패턴은 단순합니다. AI가 새 정보를 처리할 때마다 지식을 위키 형태로 축적하게 하는 것입니다. 쿼리할 때마다 원본 문서를 다시 뒤지는 RAG와 달리, LLM Wiki는 **지식을 한 번 컴파일하고 복리로 성장**시킵니다.


이 글에서는 LLM Wiki의 구조와 동작 방식을 해부하고, 커뮤니티가 실제로 어떻게 적용했는지, 그리고 직접 구축한 시스템을 소개합니다.

## LLM Wiki란 무엇인가


LLM Wiki는 세 개의 계층으로 구성됩니다.


```plain text
Raw Sources (원본 자료, 불변)
        ↓ [Ingest]
Wiki Pages (LLM이 소유하는 마크다운 파일)
        ↑↓
Schema / CLAUDE.md (행동 설정 문서)
```


**Raw Sources**는 PDF, 노트, 기사, 오디오 등 사용자가 주입하는 원본 자료입니다. LLM은 이 계층을 읽기만 하고 수정하지 않습니다.


**Wiki Pages**는 LLM이 소유하는 마크다운 파일 모음입니다. 요약 페이지, 개념 페이지, 엔티티 페이지, 비교 문서 등이 여기에 쌓입니다. Karpathy의 표현을 빌리면, "LLM이 이 계층을 완전히 소유합니다. 페이지를 만들고, 새 소스가 오면 업데이트하고, 크로스레퍼런스를 유지합니다."


**Schema**는 [CLAUDE.md](http://claude.md/) 같은 설정 문서입니다. Wiki의 구조, 컨벤션, 워크플로우를 정의해 LLM이 일관되게 행동하도록 합니다.


### RAG, Memory와 무엇이 다른가


| 방식       | 동작              | 강점            | 약점                 |
| -------- | --------------- | ------------- | ------------------ |
| RAG      | 쿼리마다 원본 문서 검색   | 높은 리콜, 최신 정보  | 합성 미축적, 사람이 읽기 어려움 |
| Memory   | 대화 중 자동 업데이트    | 자동화           | 업데이트 기준 불명확, 구조 없음 |
| LLM Wiki | 소스 처리 시 위키 업데이트 | 지식 복리 성장, 투명성 | 실시간 데이터 부적합        |


LLM Wiki의 핵심 가치는 **투명성**입니다. AI가 어떤 파일을 참조하는지 사람이 정확히 알 수 있습니다. 벡터화되어 블랙박스가 되는 RAG와 달리, 모든 지식이 읽을 수 있는 마크다운으로 존재합니다.


세 방식은 대립하지 않습니다. 높은 리콜이 필요한 영역은 RAG, 대화 맥락은 Memory, 복리로 쌓아야 할 지식은 LLM Wiki — 문제 성격에 따라 함께 사용할 수 있습니다.

## 3가지 핵심 오퍼레이션


### Ingest — 소스를 위키로 컴파일


새 소스가 들어오면 LLM은 단순히 인덱싱하지 않습니다. 소스를 읽고, 핵심 개념과 엔티티를 추출하고, 관련된 기존 페이지를 찾아 업데이트하고, 새 페이지가 필요하면 생성하고, 크로스레퍼런스를 유지하고, 인덱스와 로그를 갱신합니다.


Karpathy는 이렇게 설명합니다.

> "하나의 소스가 10~15개 위키 페이지에 영향을 줄 수 있습니다."

이것이 RAG와 결정적으로 다른 지점입니다. RAG는 소스를 쪼개 벡터로 저장합니다. LLM Wiki는 소스를 **이해해서 기존 지식 구조에 통합**합니다.


### Query — 위키에서 답을 합성하고 다시 저장


질문이 들어오면 LLM은 [INDEX.md](http://index.md/)를 보고 관련 페이지를 찾아 읽고 답변을 합성합니다. 출처를 인용하기 때문에 어디서 온 답인지 추적할 수 있습니다.


주목할 부분은 **파일백(file-back)** 메커니즘입니다. 좋은 답변, 탐색 중 발견한 새로운 연결고리는 다시 위키 페이지에 저장됩니다. 질문 자체가 지식을 성장시키는 계기가 됩니다.


### Lint — 위키의 자가 점검


주기적으로 LLM이 위키 전체를 스캔해 건강 상태를 점검합니다.

- 페이지 간 모순되는 주장
- 새로운 소스에 의해 낡아진 클레임
- 아무도 링크하지 않는 고아 페이지
- 있어야 할 크로스레퍼런스 누락

사람이 위키를 포기하는 이유는 읽기나 생각이 힘들어서가 아닙니다. **장부 기록이 지루하기 때문**입니다. LLM은 이 작업을 거의 비용 없이 수행합니다.


### [INDEX.md](http://index.md/)와 [LOG.md](http://log.md/)


[**INDEX.md**](http://index.md/)는 모든 위키 페이지를 한 줄 요약과 함께 나열하는 카탈로그입니다. LLM이 복잡한 검색 인프라 없이 관련 페이지를 빠르게 찾을 수 있게 합니다.


[**LOG.md**](http://log.md/)는 인제스트·쿼리·린트의 이력을 append-only로 기록합니다. 무엇을 언제 처리했는지 추적하고, 오래된 클레임을 감지하는 기준이 됩니다.

## 언제 써야 하고 언제 쓰지 말아야 하는가


### 적합한 경우


**지식이 복리로 쌓여야 할 때.** 개인 세컨드 브레인, 팀 자산화, 커리어 관리 — 데이터가 누적되고 연결될 때 가치가 생기는 영역입니다.


**여러 AI나 사람이 같은 지식을 참조해야 할 때.** 프로젝트 컨텍스트, 제안서 작성 패턴, 고객 데이터 — 매번 재유도하지 않고 공유 기반 지식으로 참조할 수 있습니다.


**데이터 주권이 중요할 때.** 모든 파일이 로컬 파일 시스템에 있습니다. 제공 업체에 종속되지 않고, 어떤 AI 모델로도 교체할 수 있습니다.


### 부적합한 경우


| 상황               | 이유                       | 대안          |
| ---------------- | ------------------------ | ----------- |
| 실시간 데이터 (주가, 날씨) | 위키는 인제스트 시점의 스냅샷         | 실시간 API 연동  |
| 하나도 빠지면 안 되는 업무  | 합성 과정에서 누락 가능            | RAG + 벡터 검색 |
| 일회성 자료           | 정리할 필요 없이 raw source로 충분 | 원본 그대로 보관   |
| 추상화가 불필요한 단순 검색  | 오버엔지니어링                  | 단순 파일 검색    |

## 커뮤니티 구현 사례


Karpathy의 gist에는 25개 이상의 댓글로 실제 구현 사례가 쌓였습니다. 그중 주목할 세 가지입니다.


### Shilren — 이력서 일관성 유지


이직 준비를 하다 보면 이력서, 자기소개서, 면접 답변에서 같은 경험을 다르게 서술하게 됩니다. "22개월"과 "23개월"처럼 작은 수치 불일치도 신뢰를 깎습니다.


Shilren은 경험과 성과 수치를 단일 출처 위키 페이지로 정리하고, 모든 문서를 이 파일 기반으로만 생성하도록 했습니다. 그리고 중요한 인사이트를 남겼습니다.

> "50~100k 토큰 이하에서는 RAG가 불필요합니다. 그냥 컨텍스트에 올리면 됩니다."

위키가 충분히 작을 때는 벡터 검색 없이 파일을 직접 읽히는 것으로 충분합니다.


### Electro-resonance — LLM-WIKI-MCP


위키를 MCP 서버로 서빙하는 구현입니다. `wiki_search(category, query)`, `wiki_get(path)` 같은 도구를 제공해 어떤 Claude 세션에서도 위키에 접근할 수 있습니다. 파일 경로를 알려주는 방식보다 훨씬 구조적인 크로스 프로젝트 접근이 가능합니다.


### vvvvvivekkk — 마크다운 + Git이면 충분

> "다음 세대의 지식 시스템은 벡터 DB나 채팅 히스토리가 아닙니다. 버전 관리되는 지식 시스템입니다."

마크다운과 Git만으로 운영하고, 모든 인덱스는 재생성 가능하게 설계했습니다. 복잡한 인프라 없이 데이터의 출처와 이력을 완벽하게 추적합니다.

## 직접 구축한 시스템


이론을 따라 실제로 구축해봤습니다. 두 부분으로 나뉩니다.


### wiki/ — 카테고리 격리 설계


위키가 방대해질 것을 고려해 **카테고리 격리**를 핵심으로 설계했습니다.


```plain text
E:/development/wiki/
├── career/     ← 이직·취업·면접
├── personal/   ← 개인 생활 (finance/, health/, life/)
├── tech/       ← 기술 지식 (java-spring/, frontend/, devops/)
└── ai-llm/     ← AI·LLM 특화
```


각 카테고리는 독립된 `CLAUDE.md`를 가집니다. `wiki/career/`에서 Claude Code를 열면 career 컨텍스트만 로드됩니다. AI는 해당 카테고리 파일만 읽고 씁니다. 위키가 커져도 AI가 읽어야 할 문서는 관련 카테고리로 제한됩니다.


### wiki-agent/ — 항상 살아있는 에이전트 레이어


위키 파일만으로는 부족합니다. 데이터가 자동으로 흘러들어와야 위키가 살아있게 됩니다.


```plain text
[Telegram 메시지]   [Google Calendar CronJob]
        ↓                      ↓
        └──────────┬───────────┘
                   ↓
        에이전트 (Python, 로컬 PC 상시 실행)
          Claude API → wiki/ 읽기/쓰기
                   ↓
        Notion 단방향 동기화 (사람용 뷰어)
```

- **Telegram**: 폰에서 즉흥적으로 메모나 질문을 보내면 위키가 업데이트됩니다
- **Google Calendar CronJob**: 매일 자정 일정을 가져와 `personal/pages/life/daily-log.md`를 업데이트합니다
- **Notion**: 위키(AI용)를 사람이 보기 좋은 형태로 미러링합니다

세션이 열릴 때만 동작하는 Claude Code와 달리, 이 에이전트는 이벤트가 올 때마다 깨어나 위키를 업데이트하고 다시 대기합니다.


### 향후 MCP 전환


현재는 다른 프로젝트의 [CLAUDE.md](http://claude.md/)에 위키 경로를 명시해 참조합니다. 위키가 충분히 커지면 Electro-resonance의 LLM-WIKI-MCP 방식처럼 MCP 서버로 전환할 계획입니다. 파일 구조는 그대로 유지하면서 그 위에 서비스 레이어를 얹는 방식이라 마이그레이션 비용이 낮습니다.

## 정리


| 구성 요소                                   | 역할          | 핵심 설계              |
| --------------------------------------- | ----------- | ------------------ |
| Raw Sources                             | 불변 원본 자료    | LLM은 읽기만           |
| Wiki Pages                              | 컴파일된 지식     | LLM이 소유, 복리 성장     |
| Schema ([CLAUDE.md](http://claude.md/)) | 행동 설정       | 구조·컨벤션·워크플로우 정의    |
| Ingest                                  | 소스 → 위키 통합  | 1소스가 10~15페이지 업데이트 |
| Query                                   | 위키 검색 + 파일백 | 질문이 지식을 성장시킴       |
| Lint                                    | 자가 점검       | 모순·고아·오래된 클레임 탐지   |


LLM Wiki는 AI와 함께 쌓아가는 두 번째 뇌입니다. 프롬프트가 매번 새로 시작한다면, 위키는 대화가 끝난 후에도 남습니다. Vannevar Bush가 1945년에 상상한 Memex — 연상 트레일이 있는 개인 지식 저장소 — 를 이제는 LLM이 직접 유지할 수 있습니다.


도입 비용은 낮습니다. 특별한 인프라 없이 파일 시스템만으로 시작할 수 있습니다. 한 영역에서 시작해 지식이 쌓이는 것을 확인한 뒤 확장하면 됩니다.

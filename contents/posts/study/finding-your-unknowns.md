> Thariq Shihipar(Anthropic Claude Code 팀)의 [A Field Guide to Claude Fable 5: Finding Your Unknowns](https://claude.com/blog/a-field-guide-to-claude-fable-finding-your-unknowns)(2026-07-06)를 정리한 글입니다. 예시는 저자의 [companion 페이지](https://thariqs.github.io/html-effectiveness/unknowns/)를 참고했습니다.

AI에게 작업을 맡기고 결과물을 받았는데, 어딘가 미묘하게 어긋나 있던 적이 있으신가요? 코드는 돌아가고 요구사항도 채웠는데, "이게 아닌데"라는 느낌이 드는 경우 말입니다. 흥미로운 점은 그 어긋남의 원인이 대부분 모델의 능력 부족이 아니라는 것입니다. 프롬프트에 쓰지 않은 것, 너무 당연해서 말할 생각조차 못 한 것, 나 자신도 몰랐던 것 — 그 빈틈을 AI가 최선의 추측으로 메운 결과입니다.


Anthropic Claude Code 팀의 Thariq Shihipar는 이 빈틈을 "unknowns"라고 부르고, 이렇게 주장합니다. **Claude Fable은 모델이 아니라 사용자가 unknowns를 명확히 해주는 능력이 작업 품질의 병목이 되는 첫 모델이다.** 이 글에서는 그가 제시하는 unknowns의 분류와, 구현 전·중·후 세 단계에서 unknowns를 값싸게 발견하는 기법들을 해부합니다.

## 지도는 영토가 아니다 — unknowns의 정의

> 프롬프트와 컨텍스트는 지도(map)이고, 실제 코드베이스와 제약은 영토(territory)다. 그 간극이 unknowns다.

AI에게 주는 프롬프트, 첨부한 컨텍스트, 머릿속 요구사항은 모두 "지도"입니다. 하지만 실제 작업이 일어나는 곳은 코드베이스, 제약 조건, 팀의 암묵적 합의가 존재하는 "영토"입니다. 지도가 영토를 완벽히 담을 수 없듯, 프롬프트는 항상 무언가를 빠뜨립니다.


Claude가 unknown을 만나면 멈추지 않고 **최선의 추측(best guess)으로 결정을 내립니다.** 작업 규모가 커질수록 만나는 unknown도 많아지고, 추측이 쌓일수록 결과물은 의도에서 멀어집니다. 결과물이 어긋나는 것은 버그가 아니라, 지도의 빈 곳을 모델이 스스로 채운 설계된 동작입니다.


저자는 unknowns를 네 가지로 분류합니다. 럼즈펠드의 유명한 분류(Johari Window 변형)를 개발 워크플로에 그대로 매핑한 것입니다.


| 분류               | 정의               | 예시                   |
| ---------------- | ---------------- | -------------------- |
| Known Knowns     | 프롬프트에 명시한 것      | "JWT 기반 인증을 추가해줘"    |
| Known Unknowns   | 모른다는 것을 아는 것     | "토큰 만료 정책은 어떻게 하지?"  |
| Unknown Knowns   | 너무 당연해서 말하지 않은 것 | "당연히 기존 에러 포맷을 따라야지" |
| Unknown Unknowns | 존재조차 모르는 것       | 레거시 세션 모듈과의 충돌       |


이 중 위험한 것은 아래 두 줄입니다. Known Unknowns는 질문하면 되지만, **Unknown Knowns는 내가 질문할 생각조차 안 하고, Unknown Unknowns는 질문 자체가 불가능**하기 때문입니다. 그래서 저자가 제시하는 기법들은 대부분 "AI에게 답을 시키는" 것이 아니라 "AI에게 내 빈틈을 찾게 시키는" 방향으로 설계되어 있습니다.

## Pre-implementation — unknown은 코드를 쓰기 전이 가장 싸다

> 같은 unknown이라도 발견 시점이 늦을수록 비용은 기하급수적으로 커집니다. 그래서 11개 기법 중 8개가 구현 전 단계에 몰려 있습니다.

### Blindspot Pass — AI에게 내 사각지대를 찾게 한다


낯선 코드베이스에서 작업할 때, 무엇을 물어봐야 할지조차 모르는 상태(unknown unknowns)를 정면으로 공략하는 기법입니다.


```plain text
새로운 인증 제공자를 추가하려고 하는데, 이 코드베이스의 인증 모듈에 대해
아무것도 모릅니다. blind spot pass를 해주세요 — 제가 모르고 있을
unknown unknowns를 찾아서 보고해주세요.
```


핵심은 "blind spot", "unknown unknowns"라는 표현을 직접 쓰는 것입니다. 왜 이렇게 해야 할까요? 일반적인 "코드 설명해줘"는 모델이 코드의 전부를 균등하게 요약하게 만듭니다. 반면 사각지대를 찾으라는 지시는 **내 요청과 실제 코드 사이의 불일치**에 초점을 맞추게 합니다. 요약이 아니라 진단을 시키는 것입니다.


### Brainstorm & Prototype — 봐야 아는 것은 보게 만든다


시각 디자인처럼 "원하는 것을 봐야 비로소 아는" 영역이 있습니다. 이런 영역의 unknowns는 말로 물어봐도 나오지 않습니다. 저자의 처방은 구현 전에 여러 방향을 먼저 렌더링하는 것입니다.


```plain text
이 리뷰 큐 화면을 4가지 완전히 다른 디자인 방향의 HTML 페이지로
만들어주세요. 각 방향의 트레이드오프를 상단에 정리해주세요.
```


"4가지 완전히 다른"이 중요합니다. 하나만 만들면 그것을 기준으로 미세 조정하게 되지만, 극단적으로 다른 네 개를 나란히 놓으면 **내가 무엇을 원하고 원하지 않는지가 대비를 통해 드러납니다.** Unknown Knowns("당연히 이건 아니지")가 언어화되는 순간입니다.


### The Interview — 질문의 방향을 뒤집는다


모호한 기능 명세를 다룰 때는 AI가 사용자를 인터뷰하게 합니다.


```plain text
이 기능을 구현하기 전에 저를 인터뷰해주세요. 답변에 따라 아키텍처가
바뀔 질문부터, 한 번에 하나씩 물어봐주세요.
```


두 가지 제약이 이 프롬프트의 설계 포인트입니다. **"아키텍처가 바뀔 질문부터"**는 질문의 우선순위를 비용 기준으로 정렬하고, **"한 번에 하나씩"**은 사용자가 각 질문을 대충 넘기지 못하게 만듭니다. 질문 20개를 한꺼번에 받으면 절반은 건성으로 답하게 되기 때문입니다. 인터뷰가 끝나면 결정 사항을 테이블로 남겨 이후 구현의 기준점으로 삼습니다.


### Reference — 설명 대신 소스를 가리킨다


원하는 동작을 말로 명세하는 것보다, 그 동작을 이미 구현한 코드를 가리키는 것이 훨씬 정확합니다. 다른 언어의 라이브러리여도 유효합니다.


```plain text
이 Rust 크레이트가 정확히 제가 원하는 백오프 동작을 구현하고 있습니다.
TypeScript로 포팅하기 전에, 이 크레이트의 동작을 어떻게 이해했는지
먼저 정리해서 보여주세요.
```


포인트는 포팅을 바로 시키지 않고 **이해를 먼저 검증**한다는 데 있습니다. 참조 코드에 대한 모델의 해석이 곧 새로운 지도가 되는데, 그 지도가 틀렸는지 확인하는 비용은 포팅 전이 가장 쌉니다.


### Tweakable Plan — 실행 순서가 아니라 변경 가능성으로 정렬한다


구현 계획서라면 보통 1번부터 N번까지 실행 순서로 쓰지만, 저자는 다른 정렬 기준을 제안합니다.


```plain text
구현 계획을 세워주세요. 단, 제가 가장 수정할 가능성이 높은 결정
(데이터 모델, 타입 인터페이스, UX 흐름)을 앞으로 정렬하고 강조해주세요.
```


왜 순서가 아니라 변경 가능성일까요? 계획서를 검토하는 목적이 "순서 확인"이 아니라 **"내 의도와 다른 결정을 걸러내는 것"**이기 때문입니다. 데이터 모델이나 타입 인터페이스는 한번 굳으면 바꾸는 비용이 크므로, 검토자의 주의를 그곳에 집중시키는 구조입니다.

## During implementation — 계획에서 벗어난 순간을 기록한다

> 아무리 계획해도 unknowns는 영토에 숨어 있습니다. 구현 중에 만나는 unknown은 막을 수 없으므로, 추적 가능하게 만드는 것이 목표입니다.

구현 중 Claude가 계획에 없던 상황을 만나면, 멈추는 대신 보수적인 선택을 하고 그 사실을 기록하게 합니다.


```plain text
구현 중 계획에서 벗어나야 하는 상황이 생기면, 보수적인 선택을 한 뒤
implementation-notes.md의 "Deviations" 섹션에 어떤 선택을 왜 했는지
기록해주세요.
```


이 기법의 가치는 사후 검토에 있습니다. 몇 시간짜리 자율 작업이 끝난 뒤 diff만 보면 "무엇이 바뀌었는지"는 알아도 **"어디서 모델이 추측했는지"**는 알 수 없습니다. Deviations 로그는 모델의 추측 지점만 모아 놓은 목록이므로, 검토해야 할 곳을 정확히 가리킵니다.

## Post-implementation — 내 unknowns는 팀의 unknowns가 된다

> 배포한다는 것은 다른 사람이 내 unknowns를 물려받는다는 뜻입니다.

### Pitch & Explainer — 리뷰어의 unknowns를 선제 해소한다


내가 구현 과정에서 해소한 unknowns를 리뷰어는 처음부터 다시 겪습니다. 저자는 프로토타입·스펙·구현 노트를 하나의 문서로 패키징해 리뷰어에게 전달하고, 예상되는 반론에 근거를 붙여 선제적으로 답하는 방식을 제안합니다.


### Quiz — diff를 읽었다고 이해한 것은 아니다


머지 전 마지막 관문은 자기 자신을 향합니다.


```plain text
이 변경사항을 제가 완전히 이해했는지 확인할 수 있는 리포트와 퀴즈를
만들어주세요. 각 문제는 관련 코드 섹션으로 링크해주세요.
```


코드 diff를 눈으로 훑는 것과 동작을 이해하는 것은 다릅니다. 퀴즈는 **"이해했다고 착각하는 상태"(unknown unknowns의 마지막 형태)**를 머지 전에 강제로 드러내는 장치입니다. 틀린 문제가 곧 다시 봐야 할 코드입니다.

## 정리


| 단계   | 기법                     | 겨냥하는 unknown     | 핵심 설계                 |
| ---- | ---------------------- | ---------------- | --------------------- |
| 구현 전 | Blindspot Pass         | Unknown Unknowns | 요약이 아닌 진단을 시킨다        |
| 구현 전 | Brainstorm & Prototype | Unknown Knowns   | 대비를 통해 취향을 언어화한다      |
| 구현 전 | Interview              | Known Unknowns   | 비용 큰 질문부터, 한 번에 하나씩   |
| 구현 전 | Reference              | 명세의 부정확성         | 포팅 전에 이해를 먼저 검증한다     |
| 구현 전 | Tweakable Plan         | 굳으면 비싼 결정        | 실행 순서가 아닌 변경 가능성으로 정렬 |
| 구현 중 | Implementation Notes   | 계획 밖의 추측         | 추측 지점만 모은 검토 목록을 만든다  |
| 구현 후 | Pitch & Explainer      | 리뷰어의 unknowns    | 반론에 근거를 붙여 선제 답변한다    |
| 구현 후 | Quiz                   | 이해했다는 착각         | 머지 전에 이해를 시험한다        |


모델이 강해질수록 "어떻게 시킬까"보다 "내가 무엇을 말하지 않았나"가 결과를 좌우합니다. **다음 작업은 Claude에게 답을 시키는 대신, 내 unknowns를 찾아달라는 요청으로 시작해보시기 바랍니다.**

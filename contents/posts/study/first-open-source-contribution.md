> 첫 오픈소스 기여기입니다. 작은 불편 하나를 도구로 만들다가, 같은 문제를 오픈소스에서 다시 만나 기여로 이어진 이야기입니다.
> 관통하는 한 줄은 이겁니다 — **날짜 처리 때문에 세션이 조용히 사라진다.**

---


## 기억나지 않는 한 달


매달 업무보고를 쓸 때마다 막막해지는 순간이 있습니다. 지난 한 달 동안 분명히 바쁘게 일했는데, 막상 "뭘 했더라" 적으려면 기억이 안 납니다. 커밋 로그를 뒤지고, 메신저 히스토리를 뒤지고, 그렇게 다시 복원하는 게 일입니다.


그런데 요즘은 업무 대부분을 AI와 함께 합니다. 그렇다면 그 이력은 어딘가에 세션 로그로 남아 있을 텐데, 그걸 그냥 흘려보내고 있었던 겁니다. 없으면 만들면 되겠다 싶었습니다. Claude Code 세션을 자동으로 모아 주간·월간 업무보고로 뽑아주는 도구를요.


이 글은 그 도구를 만들다가 **날짜 버그**에 걸려 넘어지고, 그 버그가 결국 별 4.4k개짜리 오픈소스에 두 건의 기여로 이어진 과정입니다. 그리고 그 과정을 저 혼자가 아니라 AI와 함께 어떻게 밟았는지에 대한 기록이기도 합니다.


---


## 없어서 만들었다 — "다 했는데요?"

> 세션은 이미 디스크에 남아 있었습니다. 문제는 그걸 아무도 읽지 않는다는 것이었죠.

그렇게 만든 플러그인이 [da-haetneundeyo](https://github.com/bangddong/da-haetneundeyo)("다 했는데요?")입니다. 구조는 단순합니다.

- **캡처** — Claude Code의 `Stop` / `SessionEnd` 훅에서 세션을 로컬 JSONL 저널에 기록합니다.
- **가공** — 쌓인 저널을 주간·월간 리포트, 워크로그, 그리고 과거를 되짚는 recall 스킬로 뽑아냅니다.
- **비용** — 런타임 의존성 0, 토큰 소비 0. 훅이 로컬 파일에 쓰는 것뿐이라 API를 부르지 않습니다.

로컬 우선(local-first), MIT 라이선스. 여기까지는 순조로웠습니다. 문제는 이걸 실제로 제 업무에 쓰기 시작하면서 드러났습니다.


---


## 도그푸딩이 잡아낸 첫 번째 날짜 버그

> 내가 커밋한 세션이 "미완료"로 뜬다. 이 위화감이 첫 번째 버그의 실마리였습니다.

직접 쓰다 보니(도그푸딩) 이상한 게 보였습니다. 분명히 코드를 커밋하고 끝낸 세션인데, 리포트에서는 그 커밋이 잡히지 않고 세션이 "미완료"처럼 추정되는 겁니다. 버그처럼 보였고, 실제로 버그였습니다.


원인은 작업 디렉토리 구조에 있었습니다. 여러 프로젝트를 담은 워크스페이스 루트(그 자체는 git 저장소가 아닌 디렉토리)를 cwd로 두고 작업하면, 하위 repo에 실제로 커밋을 해도 그 커밋이 세션에 귀속되지 않았습니다. 상위 디렉토리에서 `git log`를 봐야 커밋이 안 보이니, 도구는 "이 세션은 아무것도 커밋하지 않았다"고 판단한 거죠. 이걸 이슈 [#11](https://github.com/bangddong/da-haetneundeyo/issues/11)로 등록했습니다(2026-07-14).


해결은 이렇게 했습니다. 세션이 실제로 **편집한 파일들의 경로**를 단서로 삼아, 그 dirname을 거슬러 올라가며 진짜 repo 루트를 역추론하고, 같은 시간 창(time window) 안의 커밋을 그 세션에 귀속시키는 폴백을 넣었습니다. 테스트를 먼저 깨뜨리고(RED) 통과시키는(GREEN) TDD로 짰습니다. 이 수정은 PR [#12](https://github.com/bangddong/da-haetneundeyo/pull/12)로 머지되어(2026-07-16) v0.1.7로 릴리스됐습니다.


이건 온전히 제가 직접 발견하고 고친 버그입니다. 그리고 이때 어렴풋이 알아챘습니다 — **"날짜와 시간을 다루는 코드가 데이터를 조용히 흘린다"** 는 패턴이 있다는 걸요.

## 같은 병을 오픈소스에서 다시 만났다 — agentsview

> 내가 방금 씨름한 문제가, 남의 저장소 이슈 트래커에 이미 열려 있었습니다.

플러그인을 만들며 레퍼런스를 찾다가 오픈소스 [agentsview](https://github.com/kenn-io/agentsview)(kenn-io/agentsview)를 발견했습니다. Claude Code 세션 뷰어로, Go + Svelte 5로 짜인 별 4.4k개(4,458개), MIT 프로젝트입니다. 제가 방금 씨름하던 "세션 × 날짜"라는 도메인과 정확히 겹치는 프로젝트였습니다.


이슈 목록을 훑다가 눈에 걸리는 게 있었습니다. 다른 사용자(o-o-o-o-o)가 2026-07-10에 열어둔 이슈 [#1086](https://github.com/kenn-io/agentsview/issues/1086)이었습니다. **제가 발견한 버그가 아니라, 남이 먼저 연 이슈입니다.** 내용은 이랬습니다 — "최근 N일" 같은 rolling(구르는) 날짜 프리셋이 고정된 날짜 값으로 저장되는 바람에, 다음에 앱을 열어도 범위가 갱신되지 않고 새 세션이 사이드바와 분석에서 조용히 사라진다.


제가 플러그인에서 겪은 것과 **똑같은 병**이었습니다. 날짜를 잘못 다뤄 데이터가 소리 없이 누락되는 것. 그래서 이 열린 이슈를 골라(채택) 해결에 착수했습니다.


PR [#1194](https://github.com/kenn-io/agentsview/pull/1194) `fix(frontend): stop persisting rolling-derived session date bounds` — 규모는 +370 / −28, 6개 파일입니다. 핵심은 "고정 날짜를 저장하지 말고 **의도를 저장하라**"였습니다. rolling이라는 의도(windowDays)만 저장해두고, 실행 시점에 현재 날짜 기준으로 범위를 다시 계산하게 바꿨습니다. 이미 고정값으로 오염된 기존 설치도 버전 스탬프를 붙여 마이그레이션하도록 했고요.


이 프로젝트에는 `roborev`라는 적대적 CI 리뷰 봇이 있어서, 여러 리뷰어 관점으로 PR을 물고 늘어집니다. 5라운드에 걸쳐 지적이 나왔고, 매 라운드 지적을 반영했습니다. 마지막 라운드에서야 "No issues found."가 나왔습니다. 그리고 2026-07-21, 메인테이너 Matthew Jacobs(mjacobs)가 직접 머지하며 "Thank you!"를 남겼습니다.

## 진짜 내가 발견한 것 — 로케일 버그

> 내 머신에서만 테스트가 깨졌습니다. 그게 대표작이 됐습니다.

#1194의 필수 CI를 로컬에서 돌리던 중, 테스트 하나가 **제 머신에서만** 실패하는 걸 발견했습니다. 한국어 로케일로 설정된 Windows에서요. 다른 환경에선 멀쩡한데 제 시스템에서만 전체 스위트가 깨졌습니다.


원인은 미묘했습니다. `toLocaleString`은 **OS의 로케일**을 따라가는데, 정작 컴포넌트는 **앱의 로케일(en)** 기준으로 포맷을 기대하고 있었습니다. 그 둘이 어긋나니 비영어 시스템에서는 라벨 문자열이 달라지고, 그걸 검사하던 테스트가 통째로 실패한 겁니다. 영어권 개발자 머신에서는 절대 안 보이는 버그였습니다.


이건 이슈 [#1195](https://github.com/kenn-io/agentsview/issues/1195)로 제가 직접 등록하고(2026-07-20), PR [#1196](https://github.com/kenn-io/agentsview/pull/1196) `test(frontend): make StatusBar sync-label test locale-independent`로 고쳤습니다. +5 / −9, 딱 1개 파일입니다. 테스트가 로케일에 의존하지 않고 검사하도록 바꿨습니다.


diff는 작지만, 이건 **발견 → 이슈 등록 → 수정 → 머지**까지 온전히 제 손으로 완결한 자기 서사입니다. 그리고 "환경 의존성이 테스트를 조용히 깨뜨린다"는 것도, 결국 같은 결의 문제입니다. 보이지 않는 곳에서 무언가가 조용히 어긋나는 것.

## AI와 함께, 그러나 내가 주도한 것

> 코드를 한 줄 한 줄 제가 다 타이핑하지는 않았습니다. 그렇다고 제 기여가 아닌 것도 아닙니다.

정직하게 밝히면, 이 모든 코드를 저 혼자 손으로 짠 게 아닙니다. Claude Code와 함께 작업했습니다. 요즘 개발이 대개 그렇듯이요. 중요한 건 그 협업에서 **무엇을 제가 주도했느냐**입니다.

- **문제 정의** — "세션이 왜 사라지지"라는 위화감을 버그로 특정한 것
- **재현** — 어떤 조건(워크스페이스 루트, 한국어 로케일)에서 터지는지 좁힌 것
- **설계** — "고정값 대신 의도를 저장한다", "편집 파일에서 repo 루트를 역추론한다" 같은 해법의 방향
- **리뷰 대응** — 적대적 CI 리뷰봇의 5라운드 지적을 하나씩 판단하고 반영한 것

코드 타이핑은 AI와 나눴지만, 이 판단들은 제가 했습니다. brainstorming → TDD(RED→GREEN) → 적대적 리뷰 대응이라는 정석 워크플로를, 도구를 옆에 두고 밟은 겁니다. 기여의 본질은 타이핑이 아니라 이 판단의 연쇄에 있다고 생각합니다.

## 정리


세 개의 버그가 있었습니다. 그리고 셋은 전부 같은 얼굴을 하고 있었습니다.


| 버그                       | 어디서                | 무엇이 조용히 사라졌나                  | 결과                    |
| ------------------------ | ------------------ | ----------------------------- | --------------------- |
| 플러그인 #11 (자력 발견)         | da-haetneundeyo    | 워크스페이스 루트 cwd에서 하위 repo 커밋 귀속 | PR #12 머지 · v0.1.7    |
| agentsview #1194 (이슈 채택) | kenn-io/agentsview | rolling 날짜가 고정값으로 굳어 새 세션     | PR #1194 머지 (mjacobs) |
| agentsview #1196 (자력 발견) | kenn-io/agentsview | 로케일 불일치로 비영어 환경 테스트           | PR #1196 머지           |


첫 번째는 제가 만든 도구에서, 두 번째는 남이 연 이슈에서, 세 번째는 다시 제 손으로. 셋 다 **날짜와 환경을 다루는 코드가 데이터를 소리 없이 흘린다**는 같은 결이었습니다. 작은 불편 하나를 도구로 만든 것이, 같은 문제를 오픈소스에서 알아보는 눈이 됐고, 그게 별 4.4k개짜리 프로젝트의 첫 기여자가 되는 길로 이어졌습니다.


거창한 결심으로 오픈소스에 뛰어든 게 아닙니다. 내 문제를 풀다 보니 같은 문제가 저기에도 있었을 뿐입니다. 그리고 그 시작은, 조용히 사라지던 세션 하나였습니다.

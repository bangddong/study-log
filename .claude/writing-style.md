# Study Log 글쓰기 스타일 가이드

이 가이드는 Study Log의 모든 글에 적용되는 작성 원칙이다.
Harness Engineering 시리즈 글에서 추출한 패턴을 기반으로 한다.

---

## 글 구조

```
[도입]
독자가 직접 겪었을 법한 구체적인 장면이나 현상을 제시한다.
"이런 적 있지 않으세요?" 형태가 아닌, 독자에게 직접 묻거나 사실처럼 서술한다.
의문형은 "~한 적이 있으신가요?" 형태 (합쇼체 의문형 "~습니까" 금지).
예: "AI에게 같은 것을 반복해서 설명한 적이 있으신가요?"
예: "매 세션마다 같은 개념을 AI에게 다시 설명하고 있다면, 그건 컨텍스트가 쌓이지 않기 때문입니다."

[개념/메커니즘 해부]
- "왜?"를 항상 설명한다. 동작 방식만 나열하지 않는다.
- 이론/패턴이면 다이어그램 또는 구조 인용
- 코드가 있으면 반드시 인용. 요약만으로 끝내지 않는다.
- 소제목마다 핵심 인사이트를 한 줄로 앞에 제시한다.

[정리]
- 표로 핵심 내용 요약 (2열 이상)
- 한 줄 결론: 이 글에서 독자가 가져가야 할 것 하나

[참고]
- 출처 명시 (링크 포함)
```

---

## 문체 원칙

### 현상 → 원인 구조
버그처럼 보이는 것도 설계 의도임을 밝히는 방식으로 쓴다.

> 나쁜 예: "/effort max를 설정하면 다음 세션에 사라집니다."
> 좋은 예: "/effort max를 설정했는데 다음 세션에는 사라져 있습니다. 이것은 버그가 아닙니다. `toPersistableEffort()`가 외부 사용자에게 max를 저장하지 않도록 설계된 결과입니다."

### 독자 설정
"구현을 이해하려는 개발자." 개념을 처음 접하는 사람이 아닌, 이미 쓰고 있지만 왜 그런지 궁금한 사람을 대상으로 한다.

### 용어 처리
기술 용어는 한국어로 먼저 설명한 뒤 원어를 병기한다.
예: "컨텍스트 엔지니어링(Context Engineering)"

### 경어체
`합니다/입니다` 경어체. 구어체 섞지 않는다.

## 섹션별 작성 팁

### 도입 작성법
- 2~4문단
- 첫 문단: 독자가 겪는 상황 묘사
- 둘째 문단: 이 글에서 다룰 것 예고 (but 결론은 숨김)
- 독자를 설득해 다음 섹션을 읽게 만들어야 함

### 개념 해부 작성법
- 각 소제목은 "X는 Y다" 형태로 명확하게
- 구조/다이어그램 먼저, 설명 나중
- 구현 인용 시: 코드 블록 → 한줄 설명 → 왜 이렇게 설계했는가

### 직접 겪은 사례 — 넣지 않는다 (2026-07 결정)
- 초기 Harness Engineering 시리즈에는 "직접 겪은 사례" 섹션이 있었으나, Agentic Design Patterns 시리즈부터 **넣지 않기로 결정**했다.
- 책/자료 정리 글은 원문 내용 해부에 집중한다. 개인 프로젝트 경험은 별도의 삽질기(trouble/)나 회고 글로 분리한다.
- 삽질기·트러블슈팅 글은 경험 자체가 본문이므로 이 규칙과 무관하다.

### 정리 작성법
- 표: 개념/메커니즘 | 역할 | 핵심 설계 3열 구조 권장
- 마지막 한 줄: 이 글 전체를 관통하는 원칙 하나

## Notion 등록 속성 결정 기준

| 속성 | 결정 기준 |
|------|---------|
| Series | 글이 속한 대주제. 강의 시리즈면 강의명, 독립 주제면 "AI", "Harness Engineering" 등 |
| Slug | `study/{주제키워드}` 또는 `spring/jpa/{키워드}` — 카테고리 경로 필수 포함 |
| Tags | 글에서 실제로 다루는 기술 태그만. 연관 기술이어도 직접 다루지 않으면 제외 |
| Emoji | 글의 핵심 개념을 상징하는 이모지 1개 |
| 배포상태 | 작성 중: "작성중" → 배포 준비: "작성완료" → CI 처리 후: "배포완료" |
